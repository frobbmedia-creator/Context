Technical Deep Dive & FAQ

How does Context read my workspace without sending it to a server?
We leverage the modern File System Access API (showDirectoryPicker). When you pick a folder, your browser creates a secure, temporary local handle. Context directly maps these handles to offline web worker threads. Your codebase is parsed, filtered, and aggregated inside isolated sandbox memory—and completely flushed when you close the browser tab.
How does the Git-aware prioritization scale, and does it write to my workspace?
No write operations are performed. The app opens your filesystem workspace handles with strict read-only permissions. We locate your local .git/index file, parse its binary stream inside a web worker to map tracking flags, and cross-reference file timestamps against last-modified heuristics. This allows us to auto-assign high weights to edited code and compress raw surrounding codebases without modifying a single block of your Git tree.
How are token estimations computed without downloading a massive binary file?
Instead of forcing users to download 15MB+ Tiktoken WASM blocks, we wrote a lightweight local token parser (tokenizer.js). By scanning AST syntax densities and evaluating character weight tables across different programming languages, it calculates token distributions staying within a ±5% deviation threshold of GPT-4o/Claude-3 cl100k_base.
Firefox and Safari don't fully support the Folder Picker API. What happens?
We built a fast browser-native ZIP extraction fallback. If Context detects Safari or Firefox, it activates an offline dropzone target. Simply zip your folder, drop it in, and our background Web Worker uses native browser DecompressionStreams to inflate, index, and compress your workspace in pure sandboxed RAM.
What is a Content Security Policy (CSP), and why is yours so strict?
A Content Security Policy is a security firewall enforced by the browser. Context uses a strict header containing connect-src 'none'. This instructs the browser to immediately block and sever any attempted network outbound connection. Your code cannot leave your machine because the underlying browser platform physically prevents the network request from being made.
How are custom ignore and exclusion rules handled locally?
We automatically locate and read your local .gitignore configurations during directories mapping. Additionally, you can configure custom global rule patterns on the UI interface. All these exclusion filters are parsed and executed inside index workers so useless directories like node_modules or .direnv are completely ignored prior to computing weight priorities.
Can the application handle massive repositories without causing memory leaks?
Yes. Because our pipeline is strictly non-blocking, file processing is delegated entirely to background worker threads. Instead of caching heavy raw file strings in memory, the system indexes file-paths and dynamic modification metrics first, keeping memory allocation minimal and maintaining a smooth 60 FPS workspace UI.
What makes your output payloads safer for LLMs than standard raw code copy-pasting?
Models like Claude and Gemini are trained to parse system instructions formatted with XML tags. Context encodes every file payload within isolated XML blocks, sanitizes illegal characters that lead to parser crashes, wraps raw code inside CDATA blocks to prevent nested tags from polluting instructions, and generates checksum validation hashes so the LLM has zero confusion about file states.
